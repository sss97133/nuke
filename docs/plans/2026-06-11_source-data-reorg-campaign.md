# Source Data Reorganization Campaign

> Doctrine (Skylar, 2026-06-10): **profiles are an expression of the foundational data — the data calls them into existence.** Fix the substrate's organization and the surfaces fix themselves.

## Headline findings

### Existence audit
The 910,175-row vehicles table renders profiles indiscriminately because nothing in the stack encodes render-worthiness: is_public defaults to true and means 'was scraped' (586,364 true), a USING(true) RLS policy exposes every row to anon including 323,811 private and ~312K soft-deleted rows, the profile loader (loadVehicleData.ts) renders any id without deleted/merged checks, search filters on is_public alone, and the existing display_tier column is 99.7% default and read by nothing. Human substrate is vanishingly thin: ~520 human-linked vehicles (almost all user 0: 331 direct + 104 permissions + 4 verified), 9 profiles, and 2 claimed of 569,795 external identities - yet every unclaimed identity gets a profile page. Measured tiering of the table: ~316K dead (deleted/merged, 34.7%), ~325K tier A-dense (>=4 images + events/observations + full YMM identity, 35.7%), ~520 tier A-human, ~166K tier B comp-stub (sold price + year/make, 18.2%), ~103K tier C archive (11.4%); ~82K currently-public rows have zero images, events, and observations. Proposal: materialize an existence_tier column from these exact predicates (counters image_count/observation_count already maintained on-row), enforce centrally at RLS (replacing the USING(true) hole) plus a 3-outcome profile loader (merged->redirect, deleted->410, C->aggregate page) and tier-scoped search/sitemap, with unclaimed identities downgraded to noindex attribution stubs. No testimony deleted anywhere - dead rows redirect or 410, tier C remains queryable substrate.
