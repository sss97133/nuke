# Discourse: The Audit — What's Actually Wrong

**Date**: 2026-03-21
**Participants**: Skylar (founder, live-auditing), Claude Opus 4.6 (observing via screenshots)
**Context**: Founder walked the live site (nuke.ag) and called out specific pain points in real time. This is the raw audit — founder's words mapped to specific problems with proposed fixes.

---

## THE HOMEPAGE / FEED

### Problem: "This page is still such shit"

What I see on the feed at desktop width:

1. **IMAGE UNAVAILABLE dominates.** Out of ~18 visible cards in the grid, roughly half show "IMAGE UNAVAILABLE" with a pale blue circle placeholder. The feed's first impression is a wall of missing data. The Design Bible says "See First, Know Later" — but there's nothing to see.

2. **The data is fucked up.** This is the root cause. FB Marketplace listings are scraped but images aren't imported. The vehicles exist in the database but they're hollow — title, price, maybe mileage, no photos. The feed dutifully renders them and they look broken.

3. **"IMAGE UNAVAILABLE" as a placeholder is wrong.** It announces failure. The no-image card design (NoImageBlock) exists in the codebase — it shows specs as chips instead of a broken image state. But it's not being used consistently. Some cards show the IMAGE UNAVAILABLE circle, others might show the spec card. The inconsistency makes it worse.

### What To Do:

- **Filter by default:** The feed should default to `HAS PHOTOS` on. Vehicles without images shouldn't be in the default view. They can be found via search or by toggling the filter off. First impression must be images.
- **Fix NoImageBlock consistency:** When there's no image, ALWAYS show the spec card (YMM, price, specs as chips). Never show the broken image placeholder.
- **Image import pipeline:** FB Marketplace images need to be imported, not just linked. The scraper gets the listing URL but the image URLs expire. This is a pipeline fix, not a design fix.

## THE VEHICLE PROFILE

### Problem: "The images just always keep getting zoomed in"

The hero image section uses `object-fit: cover` which crops and zooms images to fill the container. For vehicle photos taken at various angles, distances, and aspect ratios, this means:
- A wide shot of a truck in a parking lot gets cropped to just the hood
- A portrait-orientation phone photo gets the top and bottom cut off
- An image that's already tightly framed gets over-zoomed

The FILL button in the bottom-right corner toggles fit mode, but the default is wrong. `object-fit: contain` would show the full image with letterboxing. That's less "pretty" but infinitely more useful — you can actually see the vehicle.

### What To Do:

- **Default to `contain`, not `cover`.** Show the full image. Letterbox if needed. The user came to see the vehicle, not to see a cropped fragment.
- **Remember the user's preference.** If they toggle to `fill`, remember it per-vehicle or globally.

### Problem: "The vehicle dossier is self-aggrandizing glazer. It's interrupting the header and the vehicle information."

The VEHICLE DOSSIER box:
```
┌──────────────────────────────────────────────┐
│ VEHICLE DOSSIER          PARTIAL VERIFICATION │
│                                               │
│ 1998 FORD MUSTANG GT                         │
│ [SOLD]                                        │
└──────────────────────────────────────────────┘
```

This repeats the vehicle name that's already in the badge bar above (1998 / FORD / MUSTANG GT). It takes up ~80px of vertical space to say what the badges already said. "VEHICLE DOSSIER" as a label adds nothing — the entire page IS the vehicle dossier. "PARTIAL VERIFICATION" is a status that could be a small badge, not a bordered pill in an 80px box.

It sits between the hero image and the vehicle information table, pushing the actual data down. On a laptop viewport, the user has to scroll past the hero image AND the dossier box before seeing any specs. That's the "interruption" — the dossier box is a speed bump between the visual (hero image) and the data (specs table).

### What To Do:

- **Kill the VEHICLE DOSSIER box.** The vehicle name is already in the badge bar. The verification status becomes a small badge in the badge bar (next to LIVE / FB / +842% UNDER). The SOLD status becomes a badge in the badge bar.
- **After hero image, go straight to VEHICLE INFORMATION.** No intermediate box. No repetition. Hero → specs.

### Problem: "The incomplete profile warning — minimize to a yellow triangle on the VIN line"

The incomplete profile banner:
```
┌──────────────────────────────────────────────────────────┐
│ ⚠ Incomplete profile · Missing: Missing VIN, No images  │
│                                      [Add VIN] [Claim]  │
└──────────────────────────────────────────────────────────┘
```

This is ~40px of warning banner above the hero image. For vehicles scraped from FB Marketplace, MOST profiles are "incomplete" (no VIN, limited data). So this banner appears on most vehicles. It's not an exception — it's the norm. Shouting about the norm is noise.

### What To Do:

- **Remove the banner.** Replace with inline indicators:
  - VIN row in VEHICLE INFORMATION: `VIN    —  ⚠` (the triangle is the entire warning)
  - Missing image: handled by the hero section itself (if no imported images, show NoImageBlock pattern)
- The "Add VIN" and "Claim" actions move to small inline buttons on the VIN row and the title area respectively

### Problem: "The timeline — the disappearing is killing the vibe. It should scroll up at higher speed and transform into the skinny barcode version under the subhead"

The activity timeline (the GitHub-style heatmap) shows observation activity over time. Currently it appears between the badge bar and the hero image, and as you scroll it just... disappears off the top of the viewport.

The founder's vision: as you scroll, the timeline should **compress** — the full heatmap transforms smoothly into a thin barcode-style strip that docks below the sticky badge bar. You always have temporal context. It never disappears. It just gets denser.

### What To Do:

- **Parallax scroll on the timeline.** As the page scrolls, the timeline scrolls faster (maybe 1.5x scroll speed), creating a compression effect.
- **At a threshold**, the full timeline morphs into a thin (4-6px tall) horizontal barcode strip that becomes sticky below the badge bar.
- **The barcode strip** shows the same data as the full timeline — just compressed. Dense activity = darker pixels. Sparse activity = lighter. Hover to see the date.
- **This keeps temporal context visible at all times** without consuming viewport space.

### Problem: "The subhead with all the badges should be properly sticking"

The badge bar (`1984 / CHEVROLET / K10 SWB / SCOTTSDALE / LIVE / FAIR MARKET / NV`) should be sticky — it should dock below the header as you scroll so you always know what vehicle you're looking at. Currently it scrolls away.

### What To Do:

- **Make the badge bar `position: sticky; top: 40px;`** (below the 40px header).
- This is the vehicle's identity strip. It should be permanently visible, like the header.
- The compressed timeline barcode docks below it.

### Problem: "The left column is a shit show on the vehicle profile"

On wider viewports, the vehicle profile splits into columns. The left column contains the dossier, specs, description, pricing. It works at mobile width (single column scroll). But at desktop width, the columns create layout instability — the left column's content doesn't match the right column's content in length, creating awkward whitespace or overflow.

### What To Do:

- **Evaluate whether the profile needs columns at all.** The K10 profile at mobile width (single column) actually reads better than at desktop width. The data flows linearly: badges → timeline → hero → specs → provenance → quality → description → pricing → sections → images.
- **If columns stay:** The left column should be the spec/data column (scrolls). The right column should be the image gallery (sticky or independent scroll). Two distinct content types, not a random split of the same content.

## IMAGES ACROSS THE SITE

### Problem: "Images just always keep getting zoomed in"

This is systemic, not page-specific:

1. **Feed cards** use `object-fit: cover` — images are cropped to fill 4:3 aspect ratio containers. Wide shots get butchered. Portrait shots lose top/bottom.
2. **Hero image** on vehicle profile uses `object-fit: cover` — same problem, larger.
3. **Gallery grid** on vehicle profile appears to use proper sizing (the K10's 120 images looked right), but the mix of photo orientations creates a visual chaos.

### The Design System's Answer:

The feed already has `FIT: AUTO` and image fit controls. AUTO = `contain` for dense grids (8+ columns), `cover` for standard. But the behavior should probably be:

- **Feed default:** `contain` with `--bg` colored letterbox. Show the full vehicle. Never crop.
- **Hero default:** `contain`. The hero image should show the full photo, not a zoomed crop.
- **Gallery:** `contain` in grid view. `cover` optional in "FULL" view where you want edge-to-edge images.

The `cover` mode can remain as a toggle for users who prefer it, but the default should always be `contain`. You came to see the vehicle. Cropping hides the vehicle.

## THE SCROLL PROBLEM

### Problem: "The scrollys all over are almost good but fucked"

The scroll behavior has inconsistencies:

1. **Feed scroll** works (infinite scroll with virtualization). But scroll position isn't always preserved when navigating back from a vehicle profile.
2. **Vehicle profile scroll** is a single long page. The badge bar, timeline, hero, dossier, specs, images — all in one scroll context. Nothing is sticky except the header. The user loses orientation as they scroll through 120 images.
3. **Image gallery scroll** within the vehicle profile uses its own scroll context if expanded, creating a scroll-within-scroll that competes with the page scroll.

### What To Do:

- **Sticky badge bar** (addressed above)
- **Sticky image gallery tabs** (ZONES / GRID / FULL / etc.) — when you scroll into the image section, the tab bar should dock below the badge bar
- **Feed scroll position restore** — the `useFeedScrollRestore` hook exists but may not be working reliably. Verify and fix.
- **No nested scrolls.** The image gallery should NOT have its own scroll context. It should be part of the page flow. If 120 images makes the page too long, use "Show more" pagination, not a scroll box.

## THE DATA QUALITY PROBLEM

### Problem: "Mostly the data being so fucked up"

This is the elephant. The design can be perfect and the site will still feel broken if:

- Half the feed has no images
- Vehicle titles are malformed ("CHEVROLET k10 4X4 longbed Not a C10 or K20" — that's a seller's stream of consciousness, not a normalized title)
- Prices are asking prices from FB Marketplace with no context
- Fields are empty (VIN: —, MILEAGE: —, FUEL TYPE: —, FUEL SYSTEM: —)
- Extended specs have one field (DOORS: 2) and nothing else

**Design cannot fix bad data. But design can be honest about bad data.**

### What To Do:

- **Don't show empty fields.** If FUEL TYPE is —, don't render the row. Showing "—" sixteen times in the spec table screams "we have nothing." Instead, show what we DO have, densely. The gaps are implied by their absence, not announced by their presence.
- **Normalize titles before display.** Strip seller commentary. Extract YMM. If the title is "CHEVROLET k10 4X4 longbed Not a C10 or K20", the displayed title should be "1968 CHEVROLET K10" and the seller's original text goes in a "LISTING NOTES" section.
- **Data quality score should gate the feed.** Vehicles below a threshold (say 30/100) shouldn't appear in the default feed. They exist in the database, they're searchable, but they don't get feed real estate until they have enough data to render a useful card.
- **Image-first sorting for default.** Rank vehicles with images above vehicles without, regardless of other sorting criteria. The feed should look populated, not empty.

## ORGANIZATION PAGE

### Problem: Purple cards with giant logos

The organizations page shows Facebook Marketplace as a purple card with a giant Facebook logo, ConceptCarz with a giant Z logo. These are visually dominant (large colored blocks with centered icons) in a system that's supposed to be greyscale with minimal decoration.

### What To Do:

- **Organizations should use the same card pattern as vehicles.** 2px border, greyscale, dense text. The logo can be a small favicon (14px) next to the name, not a hero banner.
- **The stats (17K VEHICLES, 3.1K VEHICLES) are the interesting data.** Lead with the stats, not the branding.

## LOGIN PAGE

### Problem: Unremarkable but honest

The login page is fine. Centered form, minimal decoration, Google/GitHub OAuth, email/phone toggle. It follows the design system. No major issues.

Minor: the "Nuke / Vehicle Provenance Engine" text above the form uses sentence case "Nuke" instead of ALL CAPS "NUKE". Should match the header wordmark.

## SEARCH PAGE

### Problem: Nearly empty

The search page shows a search input, VIN Lookup button, two suggested searches ("Barn find", "Manual transmission"), and 3 "RECENTLY LISTED" cards. On a full-width desktop viewport, this is mostly empty space.

### What To Do:

- **Search should be the command input.** The header's command input and the search page's input should be the same component. Focusing the header input on the homepage IS the search page.
- **If the search page stays as a separate route**, populate it with: trending makes, recent auction results, price movement highlights, active auctions count. The search page should be a dashboard, not a form.
