## PHASE 2: VIOLATIONS.md — THE ANTI-PATTERN LOOKUP TABLE

**File:** `docs/library/technical/design-book/VIOLATIONS.md`

This is the most operationally important file in the encyclopedia. Any agent, before writing UI code, should check this table.

### Format for Every Entry

```
### VIOLATION: [short name]
**Temptation:** Why an agent or developer might do this
**Symptom:** What it looks like in code
**Impact:** What goes wrong visually or architecturally
**Detection:** How the ESLint plugin catches it (or why it doesn't)
**Fix:** Exact corrected code
```

### Required Violation Entries

Write a complete entry for EACH of the following:

---

**V-01: border-radius on any element**
- Temptation: Rounded corners are the default in every UI framework and Tailwind. They look "friendly" and "modern."
- Symptom: `border-radius: 4px`, `rounded-md`, `rounded-full`, `borderRadius: '8px'` in inline styles
- Impact: Violates the zero-radius aesthetic. The design deliberately evokes instrument panels, technical schematics, data terminals — not consumer apps.
- Detection: ESLint rule `no-border-radius` catches values > 0. `rounded-*` Tailwind classes are caught by the no-tailwind-radius rule.
- Fix: Remove entirely. No border-radius. Ever. On any element. Including buttons, inputs, cards, modals, tooltips, tags, badges.

---

**V-02: box-shadow for depth or emphasis**
- Temptation: Drop shadows communicate layering and focus state in standard web design. 
- Symptom: `box-shadow: 0 2px 8px rgba(0,0,0,0.1)`, `shadow-md`, `elevation-2`, `boxShadow: '...'` in inline styles
- Impact: Violates the flat schematic aesthetic. Depth is communicated through border weight and background contrast, not shadows.
- Detection: ESLint rule `no-box-shadow` catches any value except `none`.
- Fix: Use `border: 2px solid var(--border-strong)` for emphasis. Use `background: var(--surface-elevated)` for layering. Never use box-shadow.
- Exception: The "glow-like-headlights-in-fog" effect (Stewart-Warner luminescence) is implemented as a specific CSS animation on specific elements — it is NOT a box-shadow. It uses `opacity` and `background` transitions. See `TOKENS.md` for the exact implementation.

---

**V-03: Hardcoded hex colors**
- Temptation: Copying a hex value from a design mockup or a known color directly into JSX/CSS.
- Symptom: `color: '#2a2a2a'`, `background: '#f5f5f5'`, `style={{ color: '#bdbdbd' }}`, CSS with literal hex values
- Impact: Dark mode breaks. Theming fails. The token system is bypassed, making future changes require global search-replace instead of a single variable update.
- Detection: ESLint rule `no-hardcoded-hex` catches hex patterns in JSX style props and CSS-in-JS.
- Fix: ALWAYS use CSS variables. `color: var(--text)` not `color: '#2a2a2a'`. `background: var(--bg)` not `background: '#f5f5f5'`. See TOKENS.md for the full mapping.

---

**V-04: Wrong font family**
- Temptation: Using system-ui, Inter, -apple-system, Roboto, or any named typeface because it "looks better" or because a component library imports its own font.
- Symptom: `font-family: 'Inter'`, `font-family: system-ui`, `font-family: -apple-system`, `font-family: 'Helvetica Neue'`
- Impact: The Arial choice is deliberate and philosophical. Arial has no era, no personality, no associations — it disappears cognitively and lets data be the signal. Any other font introduces visual noise.
- Detection: ESLint rule `no-custom-fonts` catches font-family values other than Arial and Courier New.
- Fix: 
  - UI text, labels, navigation: `font-family: var(--font-ui)` → Arial, sans-serif
  - Machine data, VINs, codes, numbers, timestamps: `font-family: var(--font-data)` → Courier New, Courier, monospace
  - Never import a Google Font. Never reference a variable font. Never use `system-ui`.

---

**V-05: Font size too large**
- Temptation: 14px-16px feels "readable." Accessibility guidelines suggest 14px minimum. Modern design systems use 14-16px body text.
- Symptom: `font-size: 14px` on labels, `text-sm` resolving to 14px, `font-size: 1rem`
- Impact: The design uses 8-11px for a reason — information density, the instrument-panel aesthetic, treating the interface as a data terminal not a reading surface. Large fonts reduce data density and make the UI feel like a consumer app.
- Detection: The ESLint plugin warns on font-size > 13px in UI contexts (check the actual rule in eslint-plugin-design-system.js).
- Fix:
  - Column headers, category labels: 8px (`var(--text-xs)`)
  - Secondary labels, metadata: 9px (`var(--text-xs)` or `var(--text-sm)`)
  - Primary data, names, values: 11px (`var(--text-sm)`)
  - Headings and section titles: 12px (`var(--text-base)`)
  - Large display text (rare): 14px (`var(--text-md)`)
  - Never use 16px or larger in data contexts.

---

**V-06: Gradients (linear-gradient, radial-gradient)**
- Temptation: Gradients add visual polish and depth. They're used for backgrounds, button states, and header treatments.
- Symptom: `background: linear-gradient(...)`, `background: radial-gradient(...)`, `gradient-to-r` Tailwind utilities
- Impact: Gradients are decorative. This design is anti-decorative. Every pixel serves information function.
- Detection: ESLint rule `no-linear-gradient` catches gradient usage.
- Fix: Flat solid color only. Use `background: var(--surface)` or `background: var(--bg)`. If you need to indicate hierarchy, use `var(--surface-elevated)`.

---

**V-07: Colored borders**
- Temptation: Using accent colors on borders to draw attention. Blue for active state, red for error, green for success.
- Symptom: `border: 1px solid blue`, `border-color: #ef4444`, `borderColor: 'var(--accent-gulf-blue)'`
- Impact: Racing accent colors are easter eggs, not functional UI states. Using them for borders or emphasis violates their sacred scarcity and makes the UI look like a themed app instead of an instrument panel.
- Detection: Code review only (ESLint does not catch semantic misuse of tokens).
- Fix: 
  - Default borders: `border: 2px solid var(--border)`
  - Emphasized borders: `border: 2px solid var(--border-strong)`
  - Active state indication: heavier border weight, not color change
  - Error states: `border: 2px solid var(--border-strong)` + error text label. No red borders.
  - Accent colors are ONLY used for: livery swatches, badge accent indicators (exact spec in 02-components.md), deliberate easter-egg moments. Never as general UI state indicators.

---

**V-08: Tailwind utility classes for any of the above**
- Temptation: Tailwind is already in the project. `rounded-sm` feels less bad than `border-radius: 4px`.
- Symptom: `className="rounded-md shadow-lg bg-blue-500 text-white p-4 font-sans"`
- Impact: Every one of those classes violates a rule. Tailwind utilities produce the same violations as inline styles and bypass the CSS variable system entirely.
- Detection: ESLint catches specific Tailwind patterns. But many slip through.
- Fix: Tailwind can be used for spacing utilities (`p-`, `m-`, `gap-`, `flex`, `grid`) ONLY when they map to the 4px grid. For colors, typography, borders, shadows, and border-radius — use CSS variables and the design system classes only.

---

**V-09: Empty states that are dead ends**
- Temptation: Render "No data available" or a spinner with no affordance when a query returns empty.
- Symptom: `<div>No vehicles found.</div>`, a loading skeleton with no fallback, an empty array mapped to nothing
- Impact: Empty states are the most important UX moment — the user has a need the system can't currently fulfill. Dead-end empty states cause abandonment. The design spec requires every empty state to offer the next action.
- Detection: Code review only.
- Fix: Every empty state must:
  1. Acknowledge the gap without apologizing
  2. Explain why (briefly)
  3. Offer a specific action (import data, search differently, connect a source)
  4. Use the correct BadgePortal depth = 0 state (see 02-components.md for the empty badge spec)

---

**V-10: Loading states that flash or stutter**
- Temptation: Using `React.lazy()` and `Suspense` with a fallback spinner, or rendering nothing while fetching.
- Symptom: White flash on route change, "Loading module..." text, layout shift when data arrives
- Impact: The design requires immediate first paint. The instrument-panel aesthetic demands that the interface appears instantaneous. Visible loading breaks the "machine that always knows" illusion.
- Detection: Runtime observation only.
- Fix:
  - Use skeleton screens that match the exact layout of the loaded content (same widths, same heights, grey `var(--surface)` fills)
  - Skeleton elements must have ZERO animation (no pulse, no shimmer — those introduce motion that the design doesn't support)
  - Pre-fetch critical data with TanStack Query's `prefetchQuery` on hover
  - Route-level code splitting is fine; the Suspense fallback must be a pixel-perfect skeleton, not a spinner

---

**V-11: Inline style objects for design properties**
- Temptation: Quick fixes via `style={{ borderRadius: 8, boxShadow: '0 2px 8px rgba(0,0,0,0.1)' }}`
- Symptom: `style={{ ... }}` props containing any of: borderRadius, boxShadow, background with hex, color with hex, fontFamily, gradient
- Impact: Bypasses both the CSS variable system AND the ESLint rules (some inline style rules are caught, but complex objects may slip through).
- Detection: ESLint `no-border-radius`, `no-box-shadow`, `no-hardcoded-hex` rules catch many cases.
- Fix: Move all design properties to CSS classes using design system tokens. If a property cannot be expressed as a CSS class, create a new utility class in `unified-design-system.css`. Never use inline styles for design properties.

---

**V-12: Third-party component default styles leaking**
- Temptation: Importing `<DatePicker>`, `<Select>`, `<Modal>`, `<Tooltip>` from a component library and using default styles.
- Symptom: Rounded corners appearing on dropdowns, blue focus rings, grey shadows on modals, default Recharts styling
- Impact: Every third-party component ships with its own design system. Theirs violates ours.
- Detection: Visual inspection only.
- Fix: See Chapter 06 (Third-Party Components) for component-by-component override instructions. The general pattern:
  1. The global `* { border-radius: 0 !important; box-shadow: none !important }` catches most cases
  2. For color violations, wrap the component in a CSS class that overrides using CSS variables
  3. For font violations, add `font-family: var(--font-ui) !important` to the component container
  4. For components with injected `<style>` tags, use the override patterns documented in Chapter 06

---

**V-13: Badge elements not wired as BadgePortals**
- Temptation: Rendering a badge as `<span className="badge">FUEL</span>` — just a styled label.
- Symptom: Static badge elements with no click handler, no depth state, no portal behavior
- Impact: The badge IS the interaction. The badge click model is the fundamental UX primitive. A badge that is just a label is architecturally wrong, not just aesthetically wrong.
- Detection: Code review.
- Fix: Use the `<BadgePortal>` component with correct props. See 02-components.md for the complete spec including `depth`, `signalType`, `vehicleId`, `onClick` handler pattern.

---

**V-14: Dark mode applied inconsistently**
- Temptation: Adding `dark:` Tailwind classes on some elements, or checking `window.matchMedia` in component code.
- Symptom: Some elements flip to dark on system dark mode, others stay light. Some elements become invisible (dark text on dark background).
- Impact: The dark mode system uses CSS custom property swaps via `[data-theme="dark"]` attribute. Any component that bypasses this creates inconsistency.
- Detection: Visual inspection in dark mode.
- Fix: See Chapter 08 (Dark Mode). Every color value must use a CSS token that has a dark-mode equivalent. Never hardcode colors. Never use `dark:` Tailwind utilities — they override the CSS variable system.

---

**V-15: 12px+ font size on data labels and column headers**
- Temptation: Labels at 12px feel "minimum accessible size."
- Symptom: Column headers at 12px, badge labels at 12px, metadata at 12px
- Impact: The 8-9px ALL CAPS label is a signature of the design. It communicates: this is a data instrument, not a website. Larger labels make it feel like a generic dashboard.
- Fix: ALL CAPS labels for columns, categories, section headers: `font-size: var(--text-xs)` (8px), `text-transform: uppercase`, `letter-spacing: 0.08em`, `font-family: var(--font-ui)`.

---

**V-16: Decorative icons or emoji**
- Temptation: Using a check mark icon for success, a warning triangle for errors, emoji for empty states.
- Symptom: `<CheckCircleIcon />`, `⚠️`, `🚗`, decorative SVG icons
- Impact: The design is text-first, data-first. Icons add visual noise unless they carry precise semantic meaning (e.g., a gauge icon for a sensor reading).
- Fix: Replace decorative icons with ALL CAPS text labels: `SUCCESS`, `WARNING`, `ERROR`. For empty states, use structured text, not illustrations. If an icon is truly necessary (maps, sensor readings), use minimal monochrome SVG aligned to the 4px grid.

---

**V-17: Padding or margin not on the 4px grid**
- Temptation: Using `padding: 10px` or `margin: 6px` for "visual balance."
- Symptom: Any padding/margin value not divisible by 4: `3px`, `5px`, `6px`, `7px`, `10px`, `14px`
- Detection: No automated detection.
- Fix: ALL spacing must be multiples of 4: `4px`, `8px`, `12px`, `16px`, `20px`, `24px`, `32px`, `40px`, `48px`. Use spacing tokens: `var(--space-1)` through `var(--space-6)` etc. as defined in the CSS.

---

**V-18: Z-index values not from the z-index scale**
- Temptation: `z-index: 9999` to "make sure" a modal is on top.
- Symptom: Arbitrary z-index values: `z-index: 100`, `z-index: 9999`, `z-index: 50`
- Fix: Use the z-index scale defined in `unified-design-system.css`. Document each layer in TOKENS.md. Never use values not in the scale.

---

**V-19: Flex/Grid gaps not on the 4px grid**
- See V-17. Applies equally to `gap`, `row-gap`, `column-gap`.

---

**V-20: Component library default focus rings**
- Temptation: Leaving default browser or library focus rings (blue outline, box-shadow outline).
- Symptom: Blue or black outline on focused buttons, inputs, links
- Fix: 
  ```css
  :focus-visible {
    outline: 2px solid var(--border-strong);
    outline-offset: 2px;
  }
  :focus:not(:focus-visible) {
    outline: none;
  }
  ```
  This is already in `unified-design-system.css` — verify it's not being overridden.

### New Section to Add: "Why These Decisions"

This is the most important thing an agent can understand. Document the WHY behind each foundational choice so agents can reason about edge cases correctly:

**Why Arial:**
Arial has no era. It does not say 1990s (Times New Roman), 2000s (Verdana), 2010s (Avenir), 2020s (Inter). It disappears. When the font disappears, the data becomes the signal. This is intentional. We are building a data instrument, not a brand experience.

**Why Courier New for machine data:**
Monospace type communicates "this came from a machine." VINs, codes, sensor values, database IDs — these are machine-generated tokens. Rendering them in Courier New makes their nature legible instantly, before the user reads the content. This is a cognitive shortcut that replaces the need for "CODE" labels.

**Why 8-9px type:**
The design treats the interface as a gauge cluster. Automotive instrument panels use small type because information density is more valuable than readability at arm's length. The user is leaning in. They want more data, not larger data. The 8-9px ALL CAPS label is not a compromise — it is the aesthetic statement.

**Why zero border-radius:**
Rounded corners are friendly. They say "consumer app." They say "safe," "approachable," "designed by a product team for adoption metrics." This interface is none of those things. It is a technical instrument for people who want precision. Square corners say "schematic," "data terminal," "trusted machine." The zero-radius rule is not about minimalism — it is about deliberately choosing a different vocabulary from mainstream consumer software.

**Why zero shadows:**
Shadows communicate depth through approximating light physics. Depth through light physics is a visual metaphor borrowed from physical objects. This interface is not a simulation of physical objects — it is a data representation. Depth is communicated through information hierarchy (nested badges, depth levels), not through simulated lighting.

**Why greyscale for primary UI:**
Color carries meaning. Using color for decoration (making a card "look nice" with a subtle blue tint) wastes the semantic capacity of color. In this design, color is reserved for: data states (chart series), racing livery easter eggs, and critical system alerts. The grey palette means that when color does appear, it means something.

**Why 4px spacing grid:**
Automotive design and technical schematics use strict dimensional grids. The 4px grid enforces mechanical regularity. It also happens to align with standard screen pixel grids and makes the layout feel engineered rather than approximated.

**Why 180ms cubic-bezier(0.16, 1, 0.3, 1):**
This easing curve starts fast and decelerates sharply. It mimics mechanical switch movement — a physical toggle or dial that snaps to position. The 180ms duration is perceptible but not noticeable. Slower feels sluggish; faster feels cheap. This specific curve was chosen to feel mechanical, not fluid (fluid = consumer app; mechanical = instrument).

## PHASE 7: 07-finder-model.md — THE FINDER-FIRST ARCHITECTURE

**File:** `docs/library/technical/design-book/07-finder-model.md`

This chapter explains how the design system serves the product vision. An agent who understands this will make better decisions in ambiguous situations.

### The Core Thesis

nuke.ag is a vehicle provenance engine. The underlying reality is a graph of facts about vehicles, ownership, events, and signals. The frontend is not the product — it is a window into the graph.

This means:
- The data topology leads. The UI follows.
- A screen is not a "page" — it is a view into a subset of the graph with a specific lens.
- The same data appears in multiple contexts (web app, CLI tool, MCP server, browser extension, third-party platform embeds) — the design system must work in all contexts.
- The "finder" mental model: the primary interaction is finding and exploring data. Features and apps are different lenses on the same backend.

### Design Implications

**Implication 1: Every element must earn its presence**
There is no decorative content. Every element either shows data, enables navigation, or enables action. If it does none of these, it should not exist.

**Implication 2: Hierarchy through data density, not visual decoration**
An important piece of information is shown with more data around it (context, history, related signals), not with larger text or brighter color. The signal's depth is visible through density, not decoration.

**Implication 3: The interface should feel inevitable**
When a user opens a vehicle profile, every number they might need should already be there. The design must not ask the user to click to reveal what they obviously need. Pre-fetch, pre-calculate, pre-render. The "Zero Click Anxiety" law from DESIGN_BIBLE.md operationalizes this.

**Implication 4: The badge is the fundamental unit**
The badge is the atomic element that both shows a data point AND enables depth navigation. The badge system is to nuke what the `<a>` tag is to the web. Every data point is potentially a portal. The design must reflect this.

**Implication 5: Anti-vanity metrics**
The design must not show numbers designed to make the user feel good. No "streak" badges. No "profile completion percentage" with a progress bar. The provenance score is a signal, not a gamification mechanic. Design every metric as if it's being read by an engineer deciding whether to buy this vehicle for $150,000.

### The Multi-Surface Model

The same backend (Supabase + Edge Functions) surfaces through:

1. **Web App** (this project) — the full design system applies
2. **MCP Server** — text/JSON output; no visual design, but data structure decisions affect what the web app can do
3. **CLI** — text output with structured formatting; the same data vocabulary
4. **Browser Extension** — context injection into third-party sites; the design system applies to the extension popup and injected elements
5. **Third-Party Platform Embeds** — widgets embedded in other sites; the design system's tokens must work in a foreign DOM

**Design Rule:** Never make a UI decision that only works on the web app. Every data structure, every component API, every interaction pattern should be expressible in a headless/text context as well.

**Practical example:** The BadgePortal `depth` prop maps to a depth level that can be represented as a nested JSON structure in the MCP server. When designing a new badge type, ask: "How would this appear in a CLI `--json` output?" If the answer is clear, the design is correct. If it's unclear, the design is wrong.

### Screen Architecture Principles

1. **Every screen has a primary query** — one data question it answers. If you can't state the primary query in one sentence, the screen is doing too much.

2. **Data leads, layout follows** — the layout of a screen is determined by what data needs to be shown and how it relates, not by visual convention. A vehicle profile shows the graph around a vehicle. The layout should reflect the graph.

3. **Screens are views, not destinations** — users should be able to deep-link to any state of any screen. URL parameters drive screen state. Never use component state for navigable screen states.

4. **The finder model means search is always present** — the command input in the header is always accessible. Any data element can be a starting point for a new search.

## APPENDIX A: DESIGN VOCABULARY GLOSSARY

Include a glossary section in the README (or a separate GLOSSARY.md) that defines all domain-specific terms used in the design system. This eliminates vocabulary ambiguity for future agents.

### Required Terms to Define

**Badge Portal:** The fundamental UI primitive. A rectangle showing a data point (label + value) that, when clicked, opens a depth-appropriate panel showing more context. Not a label. Not a chip. A portal.

**Badge Depth:** The depth level (0–3) of a badge portal indicating how much provenance data backs the displayed value. 0 = no data / not calculated. 1 = single source. 2 = multiple sources / cluster. 3 = fully sourced with audit trail.

**Signal:** A piece of information about a vehicle derived from a source (listing, document, event, sensor). Signals are the raw inputs to provenance calculations.

**Provenance:** The documented history of a vehicle's ownership, condition, modifications, and events. The core product. What the engine calculates. What the UI visualizes.

**Feed Card:** The card unit in list/feed views. Shows a vehicle summary with its top signals visible as badges.

**Finder Model:** The product mental model: the primary interaction is finding and exploring data. Everything is a search result. Features and apps are views into the same graph.

**Surface:** In design token context, the background color for elevated components (cards, panels). `var(--surface)` is `#ebebeb`. Not to be confused with the product surface (web, CLI, MCP).

**Session Zone:** The area in the header that shows user session state: authentication status, current context, active vehicle.

**Command Input:** The search/command bar in the header. The primary navigation affordance.

**Glow Effect (Stewart-Warner Luminescence):** The subtle glow applied to high-signal badges and active state elements. Implemented as an opacity transition, NOT a box-shadow. Named after the Stewart-Warner gauge cluster aesthetic reference.

**Anti-Vanity Metric:** A metric that reports what the data says, not what the user wants to hear. No artificial streak encouragement, no inflated progress indicators, no engagement-optimization framing.

**4px Grid:** The spacing system. All margins, padding, and layout measurements are multiples of 4px.

**ALL CAPS Labels:** The label style for column headers, category labels, and classification text. 8-9px, Arial, letter-spacing 0.08em, `var(--text-label)` color.

**Dead End Empty State:** An anti-pattern. An empty state that shows "no data" with no action offered. Explicitly forbidden by the design spec.

**Zero Click Anxiety:** One of the three design laws. Data must be visible without requiring clicks to reveal it. Pre-load, pre-calculate, pre-render. If a user has to click to see what they already know they want to see, the design has failed.

**See First, Know Later:** One of the three design laws. Show the data first. Explain it on demand. Don't make users read an explanation before seeing the data. The BadgePortal depth system operationalizes this.

**Every Data Point is a Live Badge:** The first design law. Every piece of data in the system should be displayable as a badge portal, wired to its source and depth. No static text is "just text" — it is a badge waiting to be wired.
